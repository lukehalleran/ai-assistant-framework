feat: FAISS IVFPQ index, zero-copy semantic search, knowledge search routing

FAISS index build (build_faiss_index.py):
- IVFFlat + OnDiskInvertedLists → IVFPQ (Product Quantization)
- 48 subquantizers × 8 bits = 48 bytes/vector (~32x compression)
- Full 41M-vector index fits in ~2 GB RAM, no ondisk inverted lists
- ADD_BATCH reduced 1M → 500K for lower peak memory

Semantic search memory optimization (semantic_search.py):
- Replaced full pandas DataFrame load (~33 GB for 40M rows) with
  zero-copy parquet row-group offset index built at load time
- Metadata read on-demand from parquet for just the ~8 result rows
  per FAISS search (grouped by row group, each RG read at most once)
- Total footprint: FAISS index (~2.2 GB) + embedder (~0.4 GB)
- _row_to_result() now takes dict instead of pandas Series

Wiki FAISS fallback in agentic search (controller.py):
- search_memory(wiki_knowledge) now always prefers FAISS semantic
  search (40M Wikipedia vectors) over sparse ChromaDB legacy data
- New methods: _search_wiki_faiss(), _format_wiki_faiss_results()

Knowledge search intent routing (handlers.py, web_search_trigger.py):
- New needs_knowledge field in handlers.py: keyword-based detection
  for encyclopedic/wiki queries ("explain in depth", "how does",
  "consult wikipedia", etc.), fires for 4+ word substantive queries
- needs_knowledge_search field added to WebSearchDecision and
  LLMSearchTriggerResponse with LLM prompt section
- 3-way routing: web search vs memory search vs knowledge search
  (at most one active per query)

Synthesis pipeline FAISS migration (synthesis_generator.py, synthesis_filter.py):
- SynthesisGenerator._sample_wiki_articles() now uses FAISS
  semantic_search_with_neighbors() instead of ChromaDB wiki_knowledge
  collection (which choked the machine loading 41M wiki embeddings)
- FAISS results normalized to match downstream dict shape (content,
  metadata, collection, relevance_score fields)
- SynthesisFilter Stage 3 (novelty external) rerouted: both claim
  similarity and co-occurrence sub-checks use FAISS wiki search
- _extract_similarity() → _extract_faiss_similarity(): reads cosine
  similarity directly from FAISS results instead of converting
  ChromaDB relevance_score
- wiki_collection constructor parameter removed from SynthesisFilter
- get_sampling_stats() reports wiki count from FAISS index
- ChromaDB still used for: facts sampling, synthesis_results storage,
  convergence tracking, and all other memory collections
- Tests updated: generator + calibration tests mock FAISS function
  instead of ChromaDB wiki queries (24/24 passing)

Config: active model → glm-5
