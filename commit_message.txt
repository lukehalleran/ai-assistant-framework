feat: STM cross-day recall disambiguation (24h window + daily notes injection + reference_type)

Diagnosed and fixed a class of failures where STM treats user
restatements of prior events as new occurrences, causing the main
response model to fabricate "twice in close proximity, that's a
pattern" framings from a single underlying event. Triaged via
daemon_debug.log forensics on a real session.

Two-layer fix:

LAYER 1 — STM prompt + output schema

core/stm_analyzer.py:
- Output schema gains two fields:
    reference_type ∈ {new_event, recall, clarification, correction, unclear}
    temporal_facts: list of normalized current-state facts
- Prompt includes 4 critical disambiguation rules:
    1. Same actors+action+outcome as recent context → recall, not new_event
    2. Collapse ambiguous temporal phrases ("did not sleep" + "all night"
       same morning = ONE night, not two)
    3. Don't invent counts/patterns from single occurrences
    4. "told them what happened" / "this situation" / "that thing" → recall
- Default to "recall" or "unclear" when uncertain — treating a recall
  as a new event is the more dangerous error
- Two prompt examples (new_event vs recall) so the model has concrete
  patterns to copy
- _parse_json backfills new fields for legacy/partial responses
- _empty_summary includes the new fields

core/prompt/builder.py (_assemble_prompt STM rendering):
- Renders Reference Type: line in [SHORT-TERM CONTEXT SUMMARY]
- When reference_type ∈ {recall, clarification, correction, unclear},
  appends explicit WARNING directive instructing main model not to
  count current message as separate occurrence or fabricate patterns
- Renders Resolved State: line from temporal_facts when populated

LAYER 2 — Context expansion: 24h window + daily notes injection

memory/corpus_manager.py:
- New method get_recent_within_hours(hours=24, max_count=30) returning
  time-bounded episodic slice (excludes summaries/reflections), tolerates
  missing/unparseable timestamps. Used by STM for full-day context
  instead of fixed last-N.

utils/daily_notes_generator.py:
- New module-level helpers get_daily_note_path() and read_daily_note()
  resolve all 3 vault layouts (flat / weekly-at-root / monthly/weekly).
  Pure read-only — never trigger generation or narrative refresh. Allow
  STM to read daily notes without instantiating DailyNotesGenerator.

core/stm_analyzer.py:
- New _get_recent_daily_notes_text() reads last STM_INJECT_DAILY_NOTES_DAYS
  daily notes from the vault and prepends them to STM's prompt as a
  "Recent daily notes (Daemon-generated EOD summaries)" section
- Graceful degradation: empty section when no notes available (e.g.
  session starts before catch-up thread finishes — observed 20s window)
- Module-contract docstring updated

core/context_pipeline.py (_analyze_stm):
- Prefers CorpusManager.get_recent_within_hours() when available; falls
  back to legacy get_recent_memories() for older corpus_managers / mocks
- Class-level hasattr(type(cm), ...) check (not instance) so Mock-based
  test fixtures fall through cleanly without explicit method registration

config/app_config.py:
- STM_RECENT_HOURS = 24 (new) — time window for recent-conversation slice,
  aligns with daily-notes EOD generation cycle
- STM_INJECT_DAILY_NOTES_DAYS = 2 (new) — yesterday + day-before
- STM_MAX_RECENT_MESSAGES bumped 8 → 30 (now serves as cap on the
  time-windowed slice rather than the slice size itself)

DESIGN CHOICES

- Daily notes injected into STM only (gpt-4o-mini consumer), NOT into
  the main response model's prompt. The main model still gets the
  existing [TEMPORAL GROUNDING] narrative_context section unchanged.
  No prompt-position fight, no token-budget interaction.
- narrative_context deliberately NOT injected into STM — keep STM lean,
  daily notes already give event-level detail STM needs for reference_type
  classification. narrative_context is month-scale abstraction, less
  useful for "is this a recall?" decisions.
- Daily notes read directly from disk (file path resolution via shared
  helper), not via ChromaDB obsidian_notes collection — avoids ~50ms
  ChromaDB query latency, simpler error path.
- 24h window is the leverage point: aligns with daily-notes EOD cycle,
  so anything older than 24h is covered by injected daily notes; weekly/
  monthly notes carry the load further back via normal retrieval.

LATENCY

- Daily notes block: ~1000 tokens for last 2 days (3303 + 711 bytes
  measured against real vault)
- Added STM call latency: ~100-300ms per call, often hidden by parallel
  query rewrite (STM and rewrite run via asyncio.gather in
  context_pipeline.py:311-326)
- Disk I/O: <10ms for both files combined
